# 元枢 ArcheOS 双工作台交互原型验证简报

## 1. 目标与边界

本简报把 PRD v0.7.0 的双工作台、Agent Factory、经营配置、审批、部署、异常、Capability Package、Knowledge、经营目标推进树和节点对话要求，转成由产品经理直接维护原型并组织 Leo 走查的交互合同。

| 项目 | 结论 |
|---|---|
| Layer | L1 Human Governance 的交互体验 + L2 Control & Evolution 的用户可见控制面 |
| Object | UX Prototype；只验证页面、导航、状态、权限提示和关键操作流程 |
| Responsibility | 让 Leo 不依赖开发者解释即可分别以两种身份完成关键旅程，并发现产品/架构/路线缺口 |
| Non-responsibility | 不实现正式 API、权限系统、Amazon/Ops Kit/飞书集成、凭证存储、Scheduler 或生产逻辑 |
| Source of truth | Product Brief v0.4.0；PRD v0.7.0 候选；Architecture v0.5.0 stale；本简报只细化可见交互，不改变上游语义 |
| Validation | 可点击原型 + Leo 两身份走查 + finding 记录 + 版本摘要 + 影响矩阵判定 |
| Reuse | 双角色框架、状态与走查方法跨领域可复用；Amazon 文案、假数据和 Capability Package 保持领域专用 |

## 2. 原型使用假设

- 复用 `apps/business-cockpit/` 的现有视觉、布局和交互模式，不要求重做品牌视觉。
- 原型是单一应用，提供两个工作台；切换身份不等于切换用户账号。
- v0.1 中 Leo 同时拥有两种身份，但原型必须按未来由不同用户承担来设计权限边界。
- 所有数据均为明确标记的假数据；任何“执行、验证、发布、审批、停止、恢复”只改变原型内状态。
- 原型必须可点击。静态截图、只读画板或无法完成状态变化的页面不满足验收。

## 3. 两种身份及持续可见上下文

### 3.1 平台管理者

负责：

- 管理 Capability Package、Knowledge、Skill、Policy 等领域资产；
- 使用 Agent Factory 生成、验证和发布 Agent Profile；
- 管理 Agent 版本、部署、Runtime、平台默认规则和平台级异常；
- 查看跨 Agent 影响、来源版本、证据和审计。

明确禁止：

- 代替店铺经营者选择 Workspace 经营导向或风险偏好；
- 代替经营者提交真实店铺凭证；
- 以平台身份批准店铺级 Amazon 写入改变；
- 查看任何明文秘密。

### 3.2 店铺经营者

负责：

- 选择已发布 Agent；
- 配置 Workspace 假凭证、经营导向、KPI、风险偏好和店铺级限制；
- 查看经营结果、异常、待办和证据；
- 审批影响 Amazon 写入方法、参数或范围的 Change；
- 暂停 Agent、紧急停止写入并确认恢复。

明确禁止：

- 创建或发布 Agent；
- 修改平台默认 Policy、通用 Capability Package、Skill 或跨 Workspace Knowledge；
- 管理平台级部署版本或跨 Agent 资产；
- 删除平台级审计证据。

### 3.3 持续可见栏

每个页面顶部必须始终显示：

1. 当前身份：`平台管理者` 或 `店铺经营者`；
2. 当前 Workspace：例如 `Amazon US Store`；
3. 当前权限范围摘要：例如 `平台资产与发布`、`当前店铺经营与审批`；
4. 当前环境：`交互原型 / 假数据`；
5. 身份切换入口。

身份切换后必须同时改变导航、默认首页、可操作按钮和空/错误状态；不能只换标题或隐藏一个按钮。

## 4. 双工作台信息架构

| 店铺经营者工作台 | 平台管理者工作台 |
|---|---|
| 经营驾驶舱（默认全部店铺） | 平台总览（默认首页） |
| 经营目标与护栏 | Agent Factory |
| Agent | Capability Packages |
| 凭证与连接帮助 | Knowledge |
| 经营分析 | 其他领域资产（Skill / Policy / Tool 来源） |
| 收件箱 | Agent 版本与 Change |
| 经营动态 / 决策记录 | 部署与 Runtime |
| Agent 运行与安全控制 | 平台异常与跨 Agent 影响 |
|  | 平台审计 |

现有页面迁移原则：

- `overview` 改为老板驾驶舱，默认全部已授权店铺汇总，并提供店铺/站点/时间/品类/商品/Agent 筛选。
- `performance` 改为三层“经营分析”，老板默认不看全部技术证据。
- `agents` 拆成经营者的“Agent”和管理者的 Agent Factory/版本页面；经营者端表达主要汇报对象和协作关系。
- 旧 `capabilities` 页面删除 Capability View 语义，改为真实 Capability Package 列表和详情。
- `evolution`、`runtime`、平台级 `governance`、跨 Agent `audit` 进入平台管理者工作台。
- `memory` 改为 Knowledge，展示四分类、版本、权限、分发和提升，不把 Tolaria 当产品模块。
- 原“待审批事项”“异常与待办”合并为“收件箱”；原“店铺级改变记录”改为“经营动态/决策记录”。
- “紧急停止与恢复”移出常用一级导航，放入 Agent 运行/安全控制；高风险状态在驾驶舱与收件箱显示快捷入口。

## 5. 关键旅程交互合同

### J1 双工作台导航与身份切换

| 字段 | 要求 |
|---|---|
| 起点 | 应用首次打开；默认进入店铺经营者的经营驾驶舱 |
| 点击步骤 | 观察持续可见栏 → 打开身份切换 → 选择平台管理者 → 查看平台默认首页与导航 → 尝试返回店铺经营者 |
| 期望结果 | 当前身份、Workspace、权限范围始终可见；两套导航与默认首页不同；切回后保留当前 Workspace |
| 失败/降级状态 | 身份上下文缺失时阻止有副作用操作并显示“当前身份不可验证”；仍允许查看只读帮助 |
| 权限禁止项 | 店铺经营者看不到 Agent 发布、平台默认规则和跨 Agent 版本操作；平台管理者默认流程不出现店铺凭证提交或经营偏好确认 |

### J2 Agent Factory 创建与发布确认

| 字段 | 要求 |
|---|---|
| 起点 | 平台管理者工作台 → Agent Factory |
| 点击步骤 | 点击“创建 Agent” → 选择 Amazon Ads 领域 → 选择已发布 Capability Package/Knowledge/默认规则 → 生成 Agent Profile 候选 → 查看验证报告 → 打开发布确认 → 逐项确认 → 点击“发布候选版本” |
| 期望结果 | 最终可见产物是 Agent Profile 候选和验证报告；确认页展示职责、KPI、精确资产版本、权限/审批、凭证需求、Schedule、Runtime、部署计划、限制、证据和回滚目标；发布后出现新版本状态 |
| 失败/降级状态 | 资产未发布、digest 变化、权限不足、Knowledge 不兼容或验证失败时发布按钮禁用，并显示具体阻塞项与返回修复入口 |
| 权限禁止项 | 店铺经营者不能进入创建/发布流程；平台管理者不能在此填写真实店铺凭证或替经营者选择经营导向 |

必须提供至少三个可切换场景：`全部通过`、`Capability Package digest 已变化`、`Knowledge 权限不兼容`。

### J3 经营偏好问答与默认规则

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → Agent → “配置并启用” |
| 点击步骤 | 选择开始引导 → 设置利润金额目标、利润率红线、订单量/营收/现金流护栏、风险偏好和写入限制 → 查看生效规则摘要 → 返回修改 → 确认；另走一次“跳过并采用默认规则” |
| 期望结果 | 清楚表达利润金额为主目标、利润率为红线、订单量/营收/现金流为护栏；不同护栏强度产生可见不同参数或限制；默认规则来源清楚；确认前可修改 |
| 失败/降级状态 | 回答冲突时采用更保守限制并显示冲突；缺少必须限制时不能启用写入，可保存为只读配置 |
| 权限禁止项 | Agent 不能放宽平台安全底线；平台管理者不能代替店铺经营者提交答案 |

### J4 凭证帮助与验证状态

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → 凭证与连接帮助 |
| 点击步骤 | 分别打开店铺凭证、广告访问凭证、大模型 API Key → 点击“如何获取” → 展开逐步帮助 → 输入假引用/假值 → 点击验证 → 查看权限范围和状态 → 触发错误场景 |
| 期望结果 | 三类凭证分开呈现；帮助包含获取步骤、所需权限、安全提示、验证方法和常见错误；状态至少有未配置、验证中、可用、权限不足、已过期、服务不可用；帮助关闭后回到原步骤 |
| 失败/降级状态 | 权限不足显示缺失权限和修复步骤；服务不可用允许稍后重试；关键凭证失败时只读或阻塞，不假装可运行 |
| 权限禁止项 | 只使用假数据；页面、通知、审计、URL 和导出中不得回显完整秘密；平台管理者不查看店铺明文值 |

### J5 影响写入的 Change 审批

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → 收件箱 → 待审批 → 一个“将改变 Amazon 写入参数生成逻辑”的 Change |
| 点击步骤 | 打开详情 → 比较当前/候选差异 → 查看影响对象、证据、验证范围和回滚 → 切换主方案/替代方案 → 分别演示批准、拒绝 → 查看状态和审计 |
| 期望结果 | 用户能理解“为什么改、会改什么、影响什么、如何验证、失败如何恢复”；批准只绑定所见版本、范围和方案；替代方案是可选的明确制品 |
| 失败/降级状态 | digest、作用域或写入语义变化后旧批准失效；证据不足时批准按钮禁用；拒绝后当前稳定版本继续生效 |
| 权限禁止项 | 平台管理者不能代替店铺经营者批准店铺写入改变；纯读取/分析 Change 不进入事前审批，但必须在改变记录中可见 |

### J6 双容器部署与写入协调

| 字段 | 要求 |
|---|---|
| 起点 | 平台管理者 → 部署与 Runtime → 一个已发布 Agent 版本 |
| 点击步骤 | 点击“新增部署” → 选择第二隔离容器 → 查看配置/SecretRef 需求 → 使用假引用完成自检 → 启动 → 查看两个容器状态和当前写入者 → 模拟 lease 失效/冲突 |
| 期望结果 | 两个部署的版本、配置状态、健康、写入范围、当前写入者和最近证据可见；同范围只显示一个有效写入者；第二容器能显示独立运行状态 |
| 失败/降级状态 | lease/协调不可用时新写入阻塞，读取分析标记只读降级；旧 fencing token 或重复意图显示被拒绝且不重复执行 |
| 权限禁止项 | 店铺经营者不能创建部署或切换平台版本；原型不建立真实容器、不调用真实 Writer、不复制凭证 |

### J7 异常闭环、恢复与紧急停止

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → 收件箱 → 异常；同时准备平台管理者的平台异常队列 |
| 点击步骤 | 打开店铺写入异常 → 确认收到 → 查看降级状态/证据 → 升级给平台管理者 → 切换身份处理平台根因 → 返回经营者 → 执行恢复检查 → 明确恢复；另演示“紧急停止”确认流程 |
| 期望结果 | 异常包含发生时间、范围、责任角色、状态、证据、建议动作和恢复检查；跨身份流转可追踪；写入只有经营者明确恢复后重启；紧急停止始终可到达 |
| 失败/降级状态 | 控制面/授权/凭证/证据服务故障时停止新写入；可信读取可只读降级；恢复检查未通过时禁止恢复；积压写入不自动补执行 |
| 权限禁止项 | 平台管理者不能替经营者恢复店铺自动写入；店铺经营者不能关闭未解决的平台根因 |

### J8 Capability Package 与 Knowledge 资产

| 字段 | 要求 |
|---|---|
| 起点 | 平台管理者 → Capability Packages；平台管理者 → Knowledge |
| 点击步骤 | 打开一个 Package → 查看结果契约、KPI/Measure、Skill/Tool/Knowledge/Policy 依赖、版本、状态和证据 → 切换发布/暂停假状态 → 查看受影响 Agent；打开四类 Knowledge → 查看版本/权限/分发 → 演示企业私有知识提升候选 |
| 期望结果 | Capability 是真实业务职责资产，Package 是其版本化制品；不存在 Capability View；Knowledge 四分类可区分，Tolaria 只作为 Markdown 来源标签；状态、版本、来源、证据、依赖影响可见 |
| 失败/降级状态 | Package 暂停/废弃或 Knowledge digest 变化时显示受影响 Agent 和阻塞/替换建议；未评审学习候选不可发布；Tolaria 不可用时已分发版本仍显示可用 |
| 权限禁止项 | 店铺经营者不能发布/暂停通用资产；企业私有知识不能跨 Workspace 分发；运行证据不能直接提升为正式 Knowledge |

### J9 老板驾驶舱与经营分析

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者默认首页 |
| 点击步骤 | 查看全部店铺汇总 → 依次切换店铺、站点、时间、品类、商品、Agent 筛选 → 打开利润变化 → 进入经营分析 → 从目标/OKR下钻到关键结果/护栏 → 展开经营目标推进树 → 查看 Agent 贡献和必要证据 |
| 期望结果 | 首屏显示营业额、利润金额、利润率、订单量及波动；30 秒内判断整体经营、最大变化、高优先待办和责任 Agent；经营分析保留三层结构并在第二/三层提供纵向推进树，技术证据默认收起 |
| 失败/降级状态 | 数据缺失、多币种换算或时间窗口不可比时明确标记；不输出虚假汇总或伪因果；高风险状态显示收件箱和紧急停止快捷入口 |
| 权限禁止项 | 不默认暴露 trace/digest/Tool 日志；筛选不得越过当前用户的 Workspace/店铺权限 |

### J10 Agent 关系与经营汇报

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → Agent |
| 点击步骤 | 查看 Agent 列表/关系 → 选择直接向老板汇报的 Agent → 查看负责 KPI、当前优先事项、协作/下级 Agent → 打开最近经营汇报 → 从汇报进入经营动态或收件箱事项 |
| 期望结果 | 页面不局限广告 Agent；每个 Agent 有唯一主要 KPI 责任和主要汇报对象；协作关系清楚但不转移责任；汇报使用结果、偏差、原因、动作、风险和需决定事项 |
| 失败/降级状态 | 汇报对象缺失、循环或 KPI 责任重复时显示关系异常，不伪装为有效组织；仍允许查看只读历史 |
| 权限禁止项 | 原型不发明跨 Agent 自动授权或任务委派；关系图不能替代服务端权限和 KPI 责任 |

### J11 经营目标推进树与策略详情

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → 经营分析 → 一个利润金额经营目标 |
| 点击步骤 | 展开季度经营目标 → 查看至少两个阶段成果 → 每阶段展开至少两个经营策略 → 每策略查看至少两个经营动作 → 点击经营策略 → 在详情抽屉查看策略假设、预设目标、适用范围、护栏、动作、进度、实际结果、复盘窗口、结论和完成判定 → 切换“动作已完成但结果未达标”场景 |
| 期望结果 | 树使用经营目标/阶段成果/经营策略/经营动作，不出现开发 Milestone/Issue；每层显示负责人、周期、状态、偏差、负责 Agent 和证据新鲜度；策略抽屉字段完整；动作完成不自动把策略/阶段成果标记完成 |
| 失败/降级状态 | 关系缺失、循环依赖、负责人缺失、证据过期或护栏触线时显示明确风险；仍可查看历史，但不能伪造达成状态 |
| 权限禁止项 | 原型不把树节点映射为 GitHub Issue/Milestone；点击动作不触发真实 Amazon 写入；不让树替代 KPI/护栏汇总 |

### J12 节点上下文对话与草案变更

| 字段 | 要求 |
|---|---|
| 起点 | 店铺经营者 → 经营分析 → 一个经营策略详情 |
| 点击步骤 | 查看右侧 Agent 对话框及当前策略上下文 → 请求调整预设目标或动作 → 查看“草案变更”回复、字段 diff、理由、影响、护栏、证据与审批要求 → 请求修改系统代码并查看拒绝 → 请求修改最高级目标/红线 → 查看待审批草案 → 触发一个已授权低风险动作并查看结果记录 |
| 期望结果 | 回复明确分类为解释/建议/草案变更/待审批动作/已执行结果；对话不直接授权；低风险动作进入经营动态/审计；对话、草案和结果均关联当前节点 |
| 失败/降级状态 | 上下文缺失、权限不明、Policy 冲突、证据不足或审计不可写时，不生成正式改变或外部动作，并给出允许的下一步 |
| 权限禁止项 | 不读取真实 Secret、不跨 Workspace、不修改系统源码或正式 Capability Package/Policy/Knowledge、不绕过 Change/审批、不把自然语言视为无限授权 |

```yaml
finding_id: UX-0001

severity: P0 | P1 | P2

  prototype_version: AR-UX-PROTOTYPE@0.1.0

  product_brief: no-change

```

严重度定义：

- P0：无法完成关键任务、越权可执行、秘密暴露、把假执行误认为真实执行或产品语义冲突；阻止原型批准。
- P1：任务可完成但高概率误判、误审批或需要开发者解释；必须有修复或明确延期决定。
- P2：文案、布局或低风险可用性问题；可在不改变产品事实时迭代。

## 9. 影响矩阵判定规则

| 判定 | 触发条件 | 处置 |
|---|---|---|
| `no-change` | 只涉及文案清晰度、布局、视觉层级、交互反馈或 mock 数据，且不改变角色、范围、状态、业务规则、权限和验收 | 原型内修复并更新 digest；不修改 PRD/Architecture/Roadmap |
| `PRD update` | 新发现改变用户角色、关键旅程、产品范围、状态语义、审批责任、验收标准、非范围或用户可见产品边界 | 产品经理形成 PRD 新版本候选；按摘要变化传播下游 stale |
| `Architecture stale` | 产品事实不变，但发现现有架构无法满足权限、安全、数据、契约、韧性或对象边界；或需要改变已批准架构语义 | 技术负责人通过 ADR/Architecture 新版本处理；Roadmap 随之 stale；原型不得自行规定实现 |
| `Roadmap update` | PRD 与 Architecture 均不变，但交付顺序、依赖、Milestone 覆盖、验收证据或角色安排不足 | 交付负责人更新 Roadmap；不得借 Roadmap 改产品或架构 |

若一个 finding 同时触发多项，先处理最上游：`PRD update → Architecture stale → Roadmap update → prototype fix`。
